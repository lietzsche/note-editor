# FEATURE-002: 노트 핵심 기능(Notes Core)

- 문서 버전: v1.0
- 작성일: 2026-04-11
- 연계 PRD: SR-2, SR-3, SR-4, SR-5, SR-6

## 1. 목적/범위

노트 조회/생성/수정/삭제/정렬 기능을 안정적으로 제공한다.

## 2. 사용자 시나리오

1. 사용자가 노트 목록을 조회한다.
2. 노트를 생성하고 즉시 목록에서 확인한다.
3. 노트를 수정하고 재조회로 변경을 확인한다.
4. 노트를 삭제하고 목록에서 제거됨을 확인한다.
5. 노트 순서를 변경하고 재로그인 후에도 유지됨을 확인한다.

## 3. 기능 요구사항

1. 목록 조회는 사용자 본인 노트만 반환한다.
2. 생성은 고유 ID를 반환하고 목록 반영이 즉시 가능해야 한다.
3. 수정은 최종 저장 상태가 재조회와 일치해야 한다.
4. 삭제는 명시적 액션으로만 실행되어야 한다.
5. 정렬 변경은 재조회 시 유지되어야 한다.

## 4. API/데이터 계약

1. 엔드포인트
- `GET /api/notes` (현재 구현 경로: `/api/pages`)
- `POST /api/notes` (현재 구현 경로: `/api/pages`)
- `GET /api/notes/:id` (현재 구현 경로: `/api/pages/:id`)
- `PUT /api/notes/:id` (현재 구현 경로: `/api/pages/:id`)
- `DELETE /api/notes/:id` (현재 구현 경로: `/api/pages/:id`)
- `POST /api/notes/reorder` (현재 구현 경로: `/api/pages/reorder`)

2. 데이터 필드(핵심)
- `id`, `title`, `content`, `sort_order`, `updated_at`, `owner_user_id`

## 5. 상태/예외 처리

1. 빈 목록 -> 정상 응답(빈 배열)
2. 잘못된 payload -> 검증 실패 응답
3. 존재하지 않는 노트 ID -> 조회/수정/삭제 실패 응답
4. 타 사용자 노트 접근 -> 권한 오류

## 6. 테스트 기준(TDD 포인트)

1. Red
- 빈 목록, 잘못된 payload, 존재하지 않는 ID 실패 테스트 선작성

2. Green
- CRUD/정렬 기본 흐름을 최소 구현으로 통과

3. Refactor
- 검증/권한/저장 로직을 분리하고 회귀 테스트 고정

## 7. 오픈 이슈
