# FEATURE-009: 관리자 비밀번호 초기화(Admin Password Reset)

- 문서 버전: v1.0
- 작성일: 2026-04-21
- 연계 PRD: (추가 예정)

## 1. 목적/범위

관리자(운영 담당자)가 시스템 사용자의 비밀번호를 초기화하거나 특정 비밀번호로 변경할 수 있도록 한다. 이 기능은 사용자가 비밀번호를 분실했을 때 계정 복구를 지원하거나, 보안 사고 발생 시 강제 비밀번호 변경을 위해 사용된다.

## 2. 사용자 시나리오

1. 관리자가 사용자 관리 페이지에서 특정 사용자를 조회한다.
2. 관리자가 해당 사용자의 비밀번호 초기화 또는 변경을 수행한다.
3. 시스템이 새로운 비밀번호를 생성하거나 지정된 비밀번호로 변경한다.
4. 변경된 비밀번호는 사용자에게 안전한 방식으로 전달된다 (예: 일회용 링크 또는 관리자 수동 전달).
5. 사용자는 새 비밀번호로 로그인할 수 있으며, 필요시 즉시 비밀번호를 변경할 수 있다.

## 3. 기능 요구사항

1. `관리자 인증`: 비밀번호 초기화 기능은 관리자 권한이 있는 사용자만 접근할 수 있다.
2. `사용자 조회`: 관리자가 사용자명이나 이메일 등으로 사용자를 검색할 수 있다.
3. `비밀번호 초기화`: 관리자가 선택한 사용자의 비밀번호를 시스템이 생성한 안전한 임시 비밀번호로 초기화한다.
4. `비밀번호 직접 설정`: 관리자가 직접 특정 비밀번호를 설정할 수 있는 옵션을 제공한다 (선택 사항).
5. `변경 기록`: 비밀번호 변경 시 관리자 ID, 변경 시간, 변경 방식을 감사 로그에 기록한다.
6. `사용자 알림`: 비밀번호가 변경되었음을 사용자에게 알리는 메커니즘을 제공한다 (이메일, 시스템 내 알림 등).
7. `강제 비밀번호 변경 플래그`: 초기화된 비밀번호로 로그인한 사용자에게 다음 로그인 시 비밀번호 변경을 요구할 수 있다.
8. `보안 정책 준수`: 비밀번호 정책(최소 길이, 복잡도)을 준수하는지 검증한다.
9. `API 보호`: 관리자 기능 API는 일반 사용자 접근을 차단하고 관리자 권한을 검증한다.

## 4. API/데이터 계약

1. 엔드포인트
- `GET /api/admin/users` - 사용자 목록 조회 (검색 필터 지원)
- `POST /api/admin/users/{userId}/password-reset` - 사용자 비밀번호 초기화/변경
- `GET /api/admin/audit/password-resets` - 비밀번호 초기화 기록 조회

2. 요청 데이터 구조 (비밀번호 초기화)
```json
{
  "action": "generate" | "set",
  "newPassword": "string (action이 'set'일 때만 필요)",
  "requireChangeOnNextLogin": "boolean",
  "notifyUser": "boolean"
}
```

3. 응답 데이터 구조
```json
{
  "userId": "string",
  "username": "string",
  "email": "string",
  "tempPassword": "string (action이 'generate'일 때만 반환)",
  "resetAt": "timestamp",
  "resetBy": "string (관리자 식별자)"
}
```

4. 응답 원칙
- 성공: 비밀번호 초기화 결과 반환
- 실패: `{"error":{"code":"...","message":"..."}}`

## 5. 상태/예외 처리

1. 관리자 권한 없음 -> 403 Forbidden
2. 존재하지 않는 사용자 -> 404 Not Found
3. 비밀번호 정책 위반 (직접 설정 시) -> 400 Bad Request
4. 잘못된 요청 데이터 -> 422 Unprocessable Entity
5. 감사 로그 기록 실패 -> 500 Internal Server Error (기능은 성공했으나 로깅 실패)

## 6. 테스트 기준(TDD 포인트)

1. Red
- 일반 사용자가 관리자 API 접근 시도 시 실패
- 존재하지 않는 사용자에 대한 초기화 시도 시 실패
- 비밀번호 정책을 위반하는 직접 설정 시도 시 실패

2. Green
- 관리자가 사용자 비밀번호 초기화 성공
- 관리자가 직접 비밀번호 설정 성공
- 감사 로그 정상 기록 확인
- 사용자 검색 기능 정상 동작

3. Refactor
- 관리자 권한 검증 미들웨어 분리
- 비밀번호 생성 유틸리티 모듈화
- 감사 로깅 서비스 추상화

## 7. 보안 고려사항

1. `관리자 권한 분리`: 관리자 기능은 별도의 권한 체계를 가져야 하며, 일반 사용자와 분리되어야 한다.
2. `감사 로그`: 모든 비밀번호 초기화 작업은 반드시 감사 로그에 기록되어야 한다.
3. `임시 비밀번호 보안`: 생성된 임시 비밀번호는 충분한 엔트로피를 가져야 하며, 가능한 한 짧은 시간 동안만 유효해야 한다.
4. `비밀번호 노출 최소화`: API 응답에 평문 비밀번호를 포함할 때는 HTTPS를 통해서만 전송하고, 로그에는 절대 기록하지 않는다.
5. `강제 변경 플래그`: 보안 사고 대응 시 강제 비밀번호 변경을 통해 추가 보안 계층을 제공한다.

## 8. 오픈 이슈

1. 관리자 권한 부여 방식 (고정 관리자 계정 vs 역할 기반 접근 제어)
2. 사용자 알림 방법 선택 (이메일, 시스템 내 알림, SMS 등)
3. 임시 비밀번호 유효 기간 설정
4. 대량 비밀번호 초기화 기능 필요 여부
5. 관리자 기능을 위한 전용 UI 필요 여부
