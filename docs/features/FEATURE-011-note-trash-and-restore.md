# FEATURE-011: 노트 휴지통 및 복원(Note Trash and Restore)

- 문서 버전: v0.1
- 작성일: 2026-04-24
- 연계 PRD: `docs/product/PRD.md` (후속 SR 확장 필요)

## 1. 목적/범위

현재 노트 삭제 동작은 사용자 입장에서 즉시 영구 삭제에 가깝다. 모바일 탭 오동작, 목록 정리 중 잘못된 선택, 공유/이동 직후의 실수 삭제가 발생하면 복구 안전망이 없다. 본 기능은 삭제를 1차적으로 `휴지통 이동`으로 전환해 데이터 손실 위험을 낮추고, 사용자가 명시적으로 복원 또는 영구 삭제를 선택할 수 있게 하는 것을 목표로 한다.

포함 범위:
1. 노트 soft delete 기반의 휴지통 이동
2. 휴지통 전용 목록/진입점
3. 휴지통에서의 복원
4. 휴지통에서의 영구 삭제
5. 휴지통 상태 노트의 일반 목록/검색/공유 제외

제외 범위:
1. 그룹 자체의 휴지통
2. 버전 히스토리/시점 복원
3. 자동 만료/배치 정리 정책
4. 다중 선택 일괄 복원/영구 삭제

## 2. 사용자 시나리오

1. 사용자가 노트 목록이나 편집 화면에서 노트를 삭제한다.
2. 시스템은 노트를 즉시 없애지 않고 휴지통으로 이동시키고, 일반 목록과 검색 결과에서는 숨긴다.
3. 사용자는 사이드바의 `휴지통` 진입점에서 삭제한 노트를 다시 찾아 읽기 전용으로 확인한다.
4. 사용자는 실수 삭제였음을 확인하고 `복원`을 눌러 원래 작업 흐름으로 되돌린다.
5. 사용자는 더 이상 필요 없는 노트에 대해서만 `영구 삭제`를 선택해 완전히 제거한다.
6. 휴지통에 있던 노트의 원래 그룹이 이미 삭제된 경우, 복원 시 기본 그룹(미분류)으로 안전하게 되돌린다.

## 3. 기능 요구사항

1. `삭제 동작 전환`: 일반 노트의 삭제 액션은 영구 삭제가 아니라 휴지통 이동으로 동작해야 한다.
2. `휴지통 상태 기록`: 노트는 `deleted_at` 타임스탬프로 휴지통 상태를 표현해야 하며, 활성 노트는 `deleted_at = null` 상태를 유지해야 한다.
3. `일반 목록 제외`: 휴지통 상태 노트는 전체 노트, 그룹별 목록, 기본 검색 결과, 정렬 대상에서 제외되어야 한다.
4. `휴지통 진입점`: UI에는 그룹 목록과 별도로 고정된 시스템 컬렉션 `휴지통`을 제공해야 한다. 이 항목은 이름 변경/삭제/재정렬 대상이 아니며 그룹 개수 제한에도 포함되지 않는다.
5. `휴지통 목록 규칙`: 휴지통 목록은 기본적으로 `deleted_at` 최신순으로 노출해야 한다.
6. `읽기 전용 미리보기`: 휴지통에서 열린 노트는 읽기 전용으로만 확인할 수 있어야 하며, 편집/자동 저장/공유/정렬 액션은 비활성화되어야 한다.
7. `복원 규칙`: 복원 시 원래 `group_id`가 여전히 유효하면 해당 그룹으로 복원하고, 그룹이 삭제되었거나 유효하지 않으면 기본 그룹(미분류)으로 복원해야 한다.
8. `복원 후 순서`: 복원된 노트는 대상 scope의 마지막 순서로 배치해 기존 정렬과 충돌하지 않도록 해야 한다.
9. `영구 삭제`: 영구 삭제는 휴지통 안에서만 가능해야 하며, 별도의 2차 확인을 거친 뒤 되돌릴 수 없어야 한다.
10. `공유 차단`: 공유 중이던 노트가 휴지통으로 이동하면 공개 링크는 즉시 비활성화되어야 하며, 복원 후에도 자동 재활성화되지 않아야 한다.
11. `기존 전환 보호 재사용`: 현재 편집 중인 노트를 삭제할 때 미저장 변경이 있으면 기존 저장/버리기 전환 정책을 그대로 재사용해야 한다.
12. `계정 격리 유지`: 다른 사용자의 휴지통 노트는 조회, 복원, 영구 삭제 모두 불가능해야 한다.

## 4. API/데이터 계약

1. 엔드포인트
- `GET /api/notes?trashed=only`
- `DELETE /api/notes/:id`
- `POST /api/notes/:id/restore`
- `DELETE /api/notes/:id/permanent`

2. 계약 원칙
- `DELETE /api/notes/:id`는 활성 노트를 휴지통으로 이동시키는 soft delete로 해석한다.
- `GET /api/notes`의 기본 응답은 활성 노트만 반환하고, `trashed=only`일 때만 휴지통 목록을 반환한다.
- `GET /api/notes/:id`는 소유자 기준으로 휴지통 노트도 조회 가능하되, 응답에 `deleted_at`이 포함되어 프론트엔드가 읽기 전용 상태를 판별할 수 있어야 한다.
- `POST /api/notes/:id/restore`는 복원된 노트의 최종 `group_id`, `sort_order`, `updated_at`을 반환해야 한다.
- `DELETE /api/notes/:id/permanent`는 이미 휴지통에 있는 노트에 대해서만 성공해야 한다.

3. 데이터 필드
- `notes`: `deleted_at TIMESTAMP NULL` 추가
- 노트 응답 확장: `deleted_at`
- 공유 토큰은 별도 레코드를 유지하더라도, 휴지통 상태 노트에는 접근할 수 없어야 한다.

## 5. 상태/예외 처리

1. 존재하지 않는 노트 삭제/복원/영구 삭제 -> `404 Not Found`
2. 타 사용자 노트 접근 -> 기존 권한 정책과 동일하게 `403` 또는 `404`
3. 이미 휴지통인 노트에 일반 삭제 재시도 -> idempotent 성공 또는 `409` 중 하나로 정책을 고정해야 한다. 초기 권장안은 idempotent 성공이다.
4. 활성 노트에 대한 영구 삭제 요청 -> `409 Conflict`
5. 이미 활성 상태인 노트 복원 요청 -> `409 Conflict`
6. 복원 대상 그룹이 사라진 경우 -> 성공 처리하되 기본 그룹으로 대체 복원
7. 휴지통 노트의 공개 링크 접근 -> `404 Not Found`
8. 휴지통 노트는 일반 검색, 그룹 필터, reorder payload 검증에서 모두 제외되어야 한다.
9. 휴지통 이동 성공 시 현재 편집 패널은 닫히거나 읽기 전용 휴지통 뷰로 전환되어야 하며, 저장 대기 상태는 남아 있지 않아야 한다.

## 6. 테스트 기준(TDD 포인트)

1. Red
- 삭제 후 노트가 일반 목록과 검색 결과에서 계속 보이는 실패 테스트를 먼저 작성한다.
- 공유 중인 노트를 휴지통으로 보낸 뒤 공개 링크가 계속 열리는 실패 테스트를 먼저 작성한다.
- 휴지통 노트의 원래 그룹이 삭제된 뒤 복원 시 기본 그룹으로 가지 않는 실패 테스트를 먼저 작성한다.
- 활성 노트에 대한 영구 삭제 요청이 허용되는 실패 테스트를 먼저 작성한다.

2. Green
- 삭제 시 휴지통 이동, 휴지통 목록 노출, 복원, 영구 삭제의 기본 흐름을 통과시킨다.
- 계정 격리, 공유 차단, 읽기 전용 상태 전환을 함께 통과시킨다.
- 캐시와 선택 상태가 휴지통 이동/복원 이후에도 일관되게 동작하도록 맞춘다.

3. Refactor
- 노트 가시성 필터(active vs trashed), 휴지통 상태 판별, 공유 차단 규칙을 공통 유틸로 분리한다.
- 기존 노트 캐시/선택 상태/자동 저장 직렬화 로직이 휴지통 상태를 별도 분기 없이 다룰 수 있도록 정리한다.

## 7. 오픈 이슈

1. 휴지통 보관 기간을 무기한으로 둘지, 30일 자동 정리 정책을 둘지 추후 확정이 필요하다.
2. 휴지통에서 제목/본문 검색을 지원할지, 1차에서는 단순 목록만 둘지 결정이 필요하다.
3. 복원 직후 노트를 자동으로 열지, 목록에만 되돌릴지 UX 결정이 필요하다.
4. 휴지통 개수 배지와 삭제 시각 표기를 기본 UI에 얼마나 강하게 드러낼지 디자인 검토가 필요하다.

## 8. 연계 테스트 시나리오

승격 시 `docs/testing/TEST_PLAN.md`에 아래 시나리오를 추가하는 것을 권장한다.

1. `TS-13 휴지통 이동 및 일반 목록 제외`
- 노트 삭제 후 전체 목록, 그룹 목록, 검색 결과에서 즉시 사라지고 휴지통에서만 보여야 한다.

2. `TS-14 휴지통 복원 및 그룹 fallback`
- 휴지통 노트 복원 시 원래 그룹으로 돌아가고, 원래 그룹이 삭제된 경우 기본 그룹으로 복원되어야 한다.

3. `TS-15 영구 삭제와 공유 차단`
- 공유 중이던 노트를 휴지통으로 이동하면 공개 링크가 차단되어야 하고, 휴지통에서 영구 삭제 후에는 복구 경로가 없어야 한다.
