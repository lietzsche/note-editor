# TEST-BACKLOG-002: 노트/그룹 전환 체감 속도 개선 및 캐싱

- 문서 버전: v0.1
- 작성일: 2026-04-15
- 연계 backlog: `docs/backlog/BACKLOG-002-note-loading-performance-and-caching.md`

## 1. 목적

노트/그룹 선택 시 느리게 느껴지는 문제를 개선하기 위한 성능/캐시 검증 초안을 관리한다.

## 2. 검증 포인트

1. 그룹 재선택 시 캐시 hit 경로가 체감상 더 빨라졌는지
2. 노트 재선택 시 편집기 내용이 더 빠르고 안정적으로 표시되는지
3. 캐시 때문에 오래된 데이터가 잘못 남지 않는지
4. 노트 수/본문 크기가 커졌을 때도 개선 효과가 유지되는지

## 3. 수동 시나리오

### TB-101 그룹 전환 냉/온 비교

1. 로그인 직후 그룹 `A`를 처음 선택한다.
2. 그룹 `B`를 선택한다.
3. 다시 그룹 `A`를 선택한다.
4. 각 전환에서 목록 표시 시간과 로딩 상태를 확인한다.

기대 결과:

- 첫 진입보다 재진입이 더 빠르게 느껴진다.
- 캐시 hit 시 빈 화면 대신 이전에 확인한 목록이 즉시 표시되거나, 지연이 크게 줄어든다.
- 백그라운드 갱신 후에도 잘못된 목록 치환이 없다.

### TB-102 노트 재선택 속도

1. 노트 `A`를 연다.
2. 노트 `B`를 연다.
3. 다시 노트 `A`를 연다.
4. 편집기 본문 표시 시간과 깜빡임 여부를 확인한다.

기대 결과:

- 이미 본 노트는 더 빠르게 다시 열린다.
- 본문이 늦게 나타나거나 잠깐 비어 보이는 현상이 줄어든다.

### TB-103 캐시 무효화

1. 노트 제목/본문 수정 후 목록과 상세를 다시 확인한다.
2. 노트를 다른 그룹으로 이동한 뒤 원래 그룹과 대상 그룹을 각각 확인한다.
3. 노트 삭제 후 캐시된 목록에 삭제 대상이 남는지 확인한다.

기대 결과:

- 수정/이동/삭제 후 오래된 캐시가 잘못 표시되지 않는다.
- 상세와 목록의 정보가 일관된다.

### TB-104 대량 데이터 체감 비교

1. 노트를 다수 생성한다.
2. 장문 본문이 포함된 노트도 여러 개 준비한다.
3. 그룹 전환과 노트 선택을 반복한다.

기대 결과:

- 노트 수와 본문 크기가 커져도 기존 대비 체감 속도가 개선된다.
- UI 멈춤이나 과도한 지연이 감소한다.

## 4. 자동화 후보

1. 통합 테스트
- 목록 payload 경량화 계약 검증
- 캐시 무효화 트리거 후 데이터 정합성 검증

2. 성능 계측 스크립트
- cold/warm group switch 시간
- cold/warm note open 시간
- payload 크기 비교

## 5. 승격 조건

1. 계측 방식이 합의된다.
2. 캐시 전략(메모리만/영속 저장 포함)이 확정된다.
3. 성공 기준이 체감 기준 또는 수치 기준 중 하나로 정리된다.
