1. 시나리오 기반 수동 검증

3. 배포 후 회귀 확인

## 3. 시나리오

### TS-01 신규 사용자 온보딩 (SR-1, SR-3)

1. 회원가입 수행
2. 로그인 수행
3. 노트 1건 생성

기대 결과:
- 가입/로그인 성공
- 생성된 노트가 목록에 즉시 반영

### TS-02 노트 CRUD (SR-2, SR-3, SR-4, SR-5)

1. 노트 3건 생성
2. 노트 1건 수정
3. 노트 1건 삭제
4. 목록 재조회

기대 결과:
- 수정 내용이 재조회 시 일치
- 삭제 노트 미노출

### TS-03 노트 정렬 (SR-6)

1. 노트 순서 변경
2. 목록 재조회
3. 재로그인 후 목록 재조회

기대 결과:
- 변경된 순서 유지

### TS-04 계정 격리 (SR-2, SR-7)

1. `user_a`에서 노트 생성
2. `user_b` 로그인 후 `user_a` 노트 접근 시도

기대 결과:
- `user_b`는 `user_a` 노트 조회/수정/삭제 불가

### TS-05 세션/보호 API (SR-1, SR-7)

1. 로그인 후 보호 API 호출 성공 확인
2. 로그아웃
3. 동일 보호 API 재호출
4. 재로그인 후 재호출
5. 브라우저 DevTools에서 세션 쿠키 속성 확인(`HttpOnly`, `SameSite`, 운영환경 `Secure`)

기대 결과:
- 로그아웃 후 호출 실패
- 재로그인 후 호출 복구
- 쿠키 보안 속성이 정책과 일치함

### TS-06 오류 처리

1. 잘못된 자격증명 로그인
2. 잘못된 payload로 노트 생성/정렬 요청
3. 로그인 실패를 제한 횟수 이상 반복해 속도 제한 응답 확인

기대 결과:
- 각 요청이 명확한 실패 응답으로 처리
- 로그인 제한 정책(예: 분당 5회 초과 시 제한)이 동작

### TS-07 그룹(폴더/카테고리) 관리 (SR-8)

1. 그룹 2개 생성(예: Work, Personal)
2. 그룹별 노트 생성
3. 그룹 필터 조회
4. 노트의 그룹 변경 후 재조회

기대 결과:
- 그룹 필터별 결과가 정확히 분리
- 그룹 변경 후 소속 정보가 재조회 시 일치
- 그룹 미사용 기본 상태에서도 노트 조회 정상 동작

### TS-08 노트 전환 독립성/무결성 (SR-9)

1. 노트 A/B 생성 후 서로 다른 본문 저장
2. A -> B -> A -> B를 빠르게 반복 전환
3. 각 전환 시점마다 본문 확인
4. 목록 이동 후 재진입 및 재로그인 후 본문 확인

기대 결과:
- 노트 A/B 내용이 서로 섞이지 않음
- 의도하지 않은 빈 본문/덮어쓰기/유실 없음
- 명시적 삭제 없이 노트 미삭제 보장

### TS-09 충돌(conflict) 처리 (SR-9)

1. 노트 A를 탭1/탭2에서 동시에 열기
2. 탭1에서 수정 후 저장
3. 탭2에서 다른 내용으로 저장 시도
4. 충돌 처리 UI에서 `덮어쓰기/취소` 중 1개 선택
5. 선택 결과에 따라 최종 본문 재조회

기대 결과:
- 충돌이 감지되고 `conflict` 상태가 노출됨
- 선택 완료 전 자동 덮어쓰기가 발생하지 않음
- `취소` 선택 시 로컬 수정 내용이 유지됨
- 선택 결과가 최종 저장 상태와 일치함

### TS-10 실시간 글자 수 표시 (SR-10)

1. 본문에 문자열 입력
2. 입력/삭제를 반복
3. 노트 A/B 전환 후 각 노트에서 다시 확인

기대 결과:
- 입력/삭제 즉시 글자 수가 갱신됨
- 노트 전환 시 현재 노트 본문 기준 글자 수가 표시됨
- 에디터 본문과 표시 글자 수가 일치함

### TS-11 노트 전체 복사 (SR-11)

1. 본문이 있는 노트에서 전체 복사 버튼 클릭
2. 붙여넣기로 복사 결과 확인
3. 빈 본문 노트에서 동일 동작 수행
4. 클립보드 실패 상황(권한 제한 등) 재현

기대 결과:
- 한 번의 액션으로 전체 본문이 복사됨
- 성공/실패 피드백이 명확히 노출됨
- 빈 본문도 복사 액션이 허용되고 성공 피드백이 표시됨

## 6. TDD 운영 방식 (권장)

1. Red
- 기능 구현 전 실패하는 테스트를 먼저 작성한다.

2. Green
- 최소 구현으로 테스트를 통과시킨다.

3. Refactor
- 테스트 통과 상태를 유지하면서 코드 구조를 개선한다.

적용 우선순위:
- SR-8 그룹 기능: 그룹 필터/소속 변경/기본 상태 테스트 선작성
- SR-9 독립성: 노트 A/B 전환 무결성 회귀 테스트 선작성
- SR-10/11 생산성: 글자 수/복사 기능 분기 테스트 선작성

## 7. 자동화 테스트 전략 (개발 시작 기준)

1. 단위 테스트
- 범위: 글자 수 계산 유틸, payload 검증, 권한 검사 유틸
- 목표: 비즈니스 규칙의 빠른 회귀 감지

2. 통합 테스트
- 범위: 인증 세션, 노트 CRUD/정렬, 그룹 필터, 충돌 응답 분기
- 목표: API 계약과 D1 연동 검증

3. E2E 테스트
- 범위: TS-01, TS-02, TS-08, TS-10, TS-11 우선 자동화
- 목표: 실제 사용자 플로우 회귀 방지

4. 운영 규칙
- PR 단계: 단위 + 통합 테스트 필수
- 배포 전: 핵심 E2E 스모크 필수

현재 자동화 스크립트:
- `npm run test:unit`: 생산성/유틸 단위 테스트
- `npm run test:integration`: Workers + D1 통합 테스트
- `npm run verify`: `typecheck + unit + integration` PR 게이트

현재 메모:
- 브라우저 기반 E2E 러너는 아직 저장소에 도입하지 않았다.
- 따라서 TS-01/02/08/10/11의 브라우저 스모크는 배포 전 수동 검증으로 유지한다.
