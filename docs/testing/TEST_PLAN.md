## 1. 테스트 방식

1. 시나리오 기반 수동 검증
- 사용자 플로우 중심으로 End-to-End 확인

2. API 동작 검증
- 인증/권한/데이터 격리 실패 케이스 포함

3. 배포 후 회귀 확인
- 배포 직후 핵심 시나리오 재실행

1. 로컬 테스트 모드: 통합 모드(단일 오리진)
2. 로컬 테스트 URL: `http://localhost:8788`
3. 테스트 대상 배포 URL: `<DEPLOY_URL>`
4. 테스트 계정
- `user_a`
- `user_b`
5. 데이터 상태
- 각 테스트 전 기본 상태(노트 0~3개) 명시

로컬 실행 커맨드:

```bash
npm run dev
```

## 3. 시나리오

### TS-01 신규 사용자 온보딩 (SR-1, SR-3)

1. 회원가입 수행
2. 로그인 수행
3. 노트 1건 생성

기대 결과:
- 가입/로그인 성공
- 생성된 노트가 목록에 즉시 반영

### TS-02 노트 CRUD (SR-2, SR-3, SR-4, SR-5)

1. 노트 3건 생성
2. 노트 1건 수정
3. 노트 1건 삭제
4. 목록 재조회

기대 결과:
- 수정 내용이 재조회 시 일치
- 삭제 노트 미노출

### TS-03 노트 정렬 (SR-6)

1. 노트 순서 변경
2. 목록 재조회
3. 재로그인 후 목록 재조회

기대 결과:
- 변경된 순서 유지

### TS-04 계정 격리 (SR-2, SR-7)

1. `user_a`에서 노트 생성
2. `user_b` 로그인 후 `user_a` 노트 접근 시도

기대 결과:
- `user_b`는 `user_a` 노트 조회/수정/삭제 불가

### TS-05 세션/보호 API (SR-1, SR-7)

1. 로그인 후 보호 API 호출 성공 확인
2. 로그아웃
3. 동일 보호 API 재호출
4. 재로그인 후 재호출

기대 결과:
- 로그아웃 후 호출 실패
- 재로그인 후 호출 복구

### TS-06 오류 처리

1. 잘못된 자격증명 로그인
2. 잘못된 payload로 노트 생성/정렬 요청

기대 결과:
- 각 요청이 명확한 실패 응답으로 처리

### TS-07 그룹(폴더/카테고리) 관리 (SR-8)

1. 그룹 2개 생성(예: Work, Personal)
2. 그룹별 노트 생성
3. 그룹 필터 조회
4. 노트의 그룹 변경 후 재조회

기대 결과:
- 그룹 필터별 결과가 정확히 분리
- 그룹 변경 후 소속 정보가 재조회 시 일치
- 그룹 미사용 기본 상태에서도 노트 조회 정상 동작

### TS-08 노트 전환 독립성/무결성 (SR-9)

1. 노트 A/B 생성 후 서로 다른 본문 저장
2. A -> B -> A -> B를 빠르게 반복 전환
3. 각 전환 시점마다 본문 확인
4. 목록 이동 후 재진입 및 재로그인 후 본문 확인

기대 결과:
- 노트 A/B 내용이 서로 섞이지 않음
- 의도하지 않은 빈 본문/덮어쓰기/유실 없음
- 명시적 삭제 없이 노트 미삭제 보장
