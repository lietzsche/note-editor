## 1. 현재 상태

`BACKLOG-002`는 1차 저위험 성능 개선 범위까지 구현과 기본 검증을 마친 상태다.

오늘 정리한 항목:
- backlog와 testing backlog 구조를 분리
- `BACKLOG-002` 문서를 기능/디자인/테스트 문서와 교차 검토해 보강
- `TEST-BACKLOG-002` 검증 포인트와 수동 시나리오 보강
- backlog 진행 우선순위를 `BACKLOG-002 -> BACKLOG-001`로 정리
- 개발 환경에서 그룹 전환/노트 열기 최소 계측 지점 추가
- 그룹별 노트 목록 메모리 캐시 + SWR 1차 구현
- 생성/수정/삭제/이동/정렬 이후 캐시 부분 갱신/무효화 규칙 반영
- `typecheck`, `unit test`, `build`, `integration test` 통과

## 2. 코드 기준 관찰 메모

현재 구현에서 확인된 핵심 포인트:
- 그룹 전환은 `loadNotes(selectedGroupId)` 재호출로 네트워크 재조회가 발생한다.
- 목록 API는 `content`를 포함한 전체 노트 데이터를 반환한다.
- 노트 클릭 자체는 추가 fetch 없이 이미 목록에 있는 노트 객체를 `openNote(note)`로 연다.

이번 반영 후 변경점:
- 같은 세션에서 이미 본 그룹은 캐시된 목록을 먼저 보여주고 백그라운드에서 재검증한다.
- 초기 진입 시 중복되던 전체 노트 fetch 1회를 제거했다.
- 생성/삭제/이동/정렬 직후에는 현재 화면을 먼저 갱신하고, 다른 scope 캐시는 무효화한다.

해석:
- 그룹 클릭이 느린 체감은 네트워크/payload 영향 가능성이 높다.
- 노트 클릭이 느린 체감은 렌더 비용, 큰 본문 state 전환, `NotesPage` 집중 상태 구조 영향 가능성이 높다.

## 3. 디자인 검토 메모

연계 디자인 문서 기준으로 다음 제약을 확인했다.

- `DESIGN_GUIDELINES`: 빈 화면보다 연속적 전환과 명확한 상태 표시가 우선
- `DESIGN_SYSTEM`: `loading/ready/error`와 `dirty/conflict` 상태 모델을 깨면 안 됨
- `COMPONENT_GUIDE`: 컨테이너와 프리젠테이셔널 컴포넌트 분리를 우선 검토해야 함

정리:
- 캐시 hit 시 즉시 표시가 가능하더라도, 편집 중인 노트의 무결성 보호가 최우선이다.
- 성능 개선은 캐시 + 상태 모델 + 렌더 구조를 함께 다뤄야 한다.
