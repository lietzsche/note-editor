# PRD: 노트 편집 서비스

- 문서 버전: v1.6
- 작성일: 2026-04-11
- 대상 프로젝트: `note-editor`
- 상태: Ready for Development

## 1. 제품 개요

본 서비스는 개인 사용자가 로그인 후 노트를 작성, 수정, 정렬, 삭제하고 계정 단위로 분리 보관하는 웹 노트 서비스다.
또한 사용자는 노트를 폴더/카테고리(그룹) 단위로 묶어 관리할 수 있어야 한다.

핵심 가치:

1. 빠른 기록: 로그인 후 즉시 작성 가능한 편집 경험
2. 데이터 신뢰성: 계정 단위 데이터 격리와 일관된 저장
3. 단순 사용성: 핵심 기능 중심의 직관적 플로우

## 2. 목표

1. 가입/로그인/노트 관리의 핵심 사용자 경험을 안정적으로 제공한다.
2. 서비스 요구사항(SR)과 인수 기준을 명확히 정의한다.
3. 서비스 문서(PRD), 테스트 문서(TEST), 인프라 문서(SPEC/DEPLOY)의 책임을 분리한다.

## 3. 비목표

1. 협업 편집, 공유 링크, 실시간 동시편집
2. 대규모 UI 리뉴얼
3. 인증 방식 전환(OAuth 등)
4. 자동 병합 기반 충돌 해결(v1 범위 외)

## 4. 사용자

1. 일반 사용자
- 개인 노트를 생성/관리하는 사용자

2. 운영 담당자
- 배포 및 운영 상태를 확인하는 담당자

## 4.1 용어 정책

1. 시스템 표준 용어는 `그룹(Group)`으로 고정한다.
2. UI 노출 용어는 `폴더` 또는 `카테고리`를 사용할 수 있다.
3. 문서/DB/API 명세에는 `group` 용어를 사용한다.

## 4.2 리소스 명명/호환 정책

1. 대외 API/문서 용어는 `note`를 표준으로 사용한다.
2. 내부 구현/레거시 경로에서 `page/pages`가 존재할 수 있으며 v1에서는 호환 alias로 허용한다.
3. 신규 구현은 `note/notes` 명명을 우선 적용한다.

## 5. 서비스 플로우

### 5.1 신규 사용자 플로우

1. 사용자는 회원가입을 수행한다.
2. 가입 성공 후 로그인한다.
3. 첫 노트를 생성하고 목록에서 확인한다.
4. 노트 수정 후 재조회 시 동일 내용을 확인한다.

### 5.2 일상 사용 플로우

1. 사용자는 로그인 후 본인 노트 목록을 조회한다.
2. 노트를 생성/수정/삭제한다.
3. 필요 시 노트 순서를 변경한다.
4. 재로그인 후에도 직전 상태가 유지되어야 한다.

### 5.3 세션 플로우

1. 로그아웃 시 보호 API 접근이 차단되어야 한다.
2. 세션 만료 상태에서 보호 API는 인증 오류를 반환해야 한다.
3. 재로그인 시 정상 접근이 복구되어야 한다.

### 5.4 권한/격리 플로우

1. 사용자 A는 사용자 B의 노트를 조회/수정/삭제할 수 없어야 한다.
2. 모든 노트 API는 사용자 경계를 기준으로 동작해야 한다.

### 5.5 그룹(Group) 관리 플로우

1. 사용자는 노트를 그룹 단위로 구성할 수 있어야 한다.
2. 그룹명은 UI에서 폴더/카테고리로 표현될 수 있다.
3. 그룹 선택 시 해당 그룹에 속한 노트만 빠르게 조회 가능해야 한다.

### 5.6 노트 전환 독립성 플로우

1. 사용자가 노트 A에서 노트 B로 이동해도 두 노트의 내용이 섞이면 안 된다.
2. 노트 전환/재진입 과정에서 저장된 내용이 임의로 삭제되면 안 된다.
3. 빠른 연속 전환(노트 A <-> B) 상황에서도 각 노트의 상태는 독립적으로 유지되어야 한다.
4. 다중 탭/세션 동시 편집 충돌 시 자동 덮어쓰기 없이 `덮어쓰기/취소` 사용자 선택으로 처리되어야 한다.

### 5.7 편집 생산성 플로우

1. 사용자는 노트 편집 중 실시간으로 글자 수를 확인할 수 있어야 한다.
2. 사용자는 현재 노트 전체 본문을 한 번의 명시적 액션으로 복사할 수 있어야 한다.
3. 복사 성공/실패는 사용자가 즉시 인지할 수 있어야 한다.

## 6. 서비스 요구사항

### SR-1 인증

1. 회원가입, 로그인, 로그아웃, 세션 조회가 제공되어야 한다.
2. 잘못된 자격증명은 명확한 실패 응답으로 처리되어야 한다.

인수 기준:
- 정상 자격증명으로 로그인 성공
- 잘못된 자격증명으로 로그인 실패
- 로그아웃 직후 보호 API 차단

### SR-2 노트 조회

1. 사용자는 본인 노트만 조회할 수 있어야 한다.
2. 노트가 없는 경우 빈 목록이 반환되어야 한다.

인수 기준:
- 계정 간 데이터 교차 조회 실패
- 빈 상태에서도 오류 없이 응답

### SR-3 노트 생성

1. 새 노트를 생성할 수 있어야 한다.
2. 생성 직후 목록/상세에서 즉시 확인 가능해야 한다.

인수 기준:
- 생성 성공 시 고유 ID 반환
- 유효하지 않은 입력은 검증 오류

### SR-4 노트 수정

1. 본인 노트의 제목/내용 수정이 가능해야 한다.
2. 수정 결과는 재조회 시 일치해야 한다.

인수 기준:
- 수정 후 재조회 일치
- 존재하지 않는 리소스 수정 실패

### SR-5 노트 삭제

1. 본인 노트 삭제가 가능해야 한다.
2. 삭제 후 목록에서 즉시 제외되어야 한다.

인수 기준:
- 삭제 직후 미노출
- 타 계정 노트 삭제 시도 실패

### SR-6 노트 정렬

1. 노트 순서 변경이 가능해야 한다.
2. 순서 변경 결과가 재조회 시 유지되어야 한다.

인수 기준:
- 순서 변경 후 유지 확인
- 잘못된 정렬 payload 오류 처리

### SR-7 보안/세션

1. 보호 API는 인증 없는 요청을 차단해야 한다.
2. 세션 정책은 일관되게 적용되어야 한다.
3. 세션 쿠키는 `HttpOnly`, `SameSite=Lax`를 기본으로 사용하고, 운영 환경은 `Secure`를 강제해야 한다.
4. 로그인 실패는 계정/IP 기준 속도 제한 정책(예: 분당 5회, 초과 시 5분 제한)을 적용해야 한다.
5. 인증 이벤트(성공/실패/로그아웃)는 감사 로그로 남겨야 한다.

인수 기준:
- 인증 없는 요청 실패
- 재로그인 후 보호 API 접근 복구
- 세션 쿠키 보안 속성이 정책과 일치함
- 로그인 과다 실패 시 제한 응답이 발생함

### SR-8 그룹(Group) 기반 관리

1. 사용자는 노트를 그룹 단위로 생성/조회/정리할 수 있어야 한다.
2. 노트는 하나의 그룹에 명확히 소속되어야 하며, 그룹 필터 조회가 가능해야 한다.
3. 그룹이 없는 기본 상태에서도 노트 작성/조회는 정상 동작해야 한다.
4. 기본 그룹(미분류)은 회원가입 시 자동 생성되어야 한다.
5. 그룹 이름은 1~40자, 사용자별 그룹 수는 최대 30개로 제한한다.

인수 기준:
- 그룹별 노트 조회 결과가 정확히 분리됨
- 그룹 변경 후 재조회 시 소속 정보가 일치함
- 그룹 미사용 상태에서도 기존 노트 기능 정상 동작
- 신규 가입 직후 기본 그룹이 존재함

### SR-9 노트 전환 시 독립성/무결성

1. 노트 간 전환 중 다른 노트 내용이 현재 노트에 섞이면 안 된다.
2. 노트 전환, 목록 이동, 재진입 과정에서 저장된 내용이 유실되면 안 된다.
3. 삭제는 명시적 삭제 액션으로만 발생해야 한다.
4. 동시 수정 충돌 발생 시 자동 덮어쓰기 없이 충돌 상태와 `덮어쓰기/취소` 선택 UI를 제공해야 한다.

인수 기준:
- 노트 A/B를 반복 전환해도 각 노트 내용 불변
- 의도하지 않은 빈 본문/덮어쓰기/유실이 발생하지 않음
- 사용자 삭제 액션 없이 노트가 사라지지 않음
- 충돌 감지 시 선택 완료 전 자동 덮어쓰기가 발생하지 않음
- 취소 선택 시 로컬 수정 내용이 유지됨

### SR-10 실시간 글자 수 표시

1. 사용자는 노트 편집 중 현재 본문의 글자 수를 실시간으로 확인할 수 있어야 한다.
2. 글자 수 표시는 편집 중 입력/삭제에 즉시 동기화되어야 한다.
3. 노트 전환 시 표시 값은 현재 선택된 노트 본문 기준으로 갱신되어야 한다.
4. 글자 수 기준은 공백/줄바꿈을 포함한 grapheme cluster(사용자 인지 문자) 기준을 사용한다.

인수 기준:
- 입력/삭제 즉시 글자 수가 반영됨
- 노트 A/B 전환 시 각 노트의 글자 수가 정확히 표시됨
- 저장 여부와 무관하게 에디터 표시 값과 일치함

### SR-11 노트 전체 복사(One-click Copy)

1. 사용자는 현재 노트 본문 전체를 명시적 액션(버튼)으로 복사할 수 있어야 한다.
2. 복사 성공 시 명확한 성공 피드백을 제공해야 한다.
3. 복사 실패 시 재시도 가능 피드백을 제공해야 한다.

인수 기준:
- 복사 버튼 1회 액션으로 전체 본문이 클립보드에 복사됨
- 성공/실패 피드백이 UI에 표시됨
- 빈 본문도 복사 액션을 허용하고 성공 피드백이 표시됨

## 7. 품질 요구사항

1. 신뢰성: 핵심 플로우 반복 실행 시 동일 결과
2. 사용성: 신규 사용자 3분 내 첫 노트 작성
3. 데이터 무결성: 노트 전환/재조회 과정에서 내용 혼선 및 유실 0건
4. 생산성: 글자 수 확인/전체 복사가 편집 흐름을 방해하지 않음
5. 운영성: 문서 기반 점검으로 배포 직후 상태 판별 가능

## 8. 성공 지표

1. 핵심 플로우(가입/로그인/CRUD/정렬/재로그인/그룹 조회) 성공률 100%
2. 계정 분리 위반 0건
3. 노트 전환 중 내용 혼선/유실 이슈 0건
4. 글자 수 표시/전체 복사 기능 회귀 0건
5. 치명 기능 장애 0건

## 11. 후속 확장 요구사항(백로그)

1. 모바일 설치형 앱(PWA) 지원
- 사용자는 모바일 브라우저에서 서비스를 홈 화면에 설치할 수 있으면 좋다.
- 설치 시 앱 이름, 앱 아이콘, 테마 색이 웹 페이지가 아니라 앱처럼 인식되도록 제공한다.
- 설치 후 진입 시 일반 브라우저 탭이 아닌 설치형 웹 앱 경험으로 열리는지 확인한다.

2. 범위 메모
- 기본 범위는 웹 앱 매니페스트, 설치용 아이콘 세트, 설치 진입 확인이다.
- iOS와 Android의 설치 진입 방식 차이는 별도 검증 항목으로 기록한다.
- 오프라인 캐시, 푸시, 백그라운드 동기화는 현재 백로그 범위에 포함하지 않는다.
