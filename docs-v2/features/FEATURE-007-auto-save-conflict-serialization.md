# FEATURE-007: 자동 저장 직렬화 및 오탐 충돌 방지

- 문서 버전: v0.1
- 작성일: 2026-04-16
- 연계 PRD: SR-9, SR-10

## 1. 목적/범위

같은 노트를 같은 페이지에서 빠르게 편집할 때 자동 저장 요청이 서로 겹치며 `conflict` 팝업이 잘못 뜨는 문제를 수정한다.

이번 범위는 `실제 다중 탭/세션 충돌`을 완화하는 기능이 아니라, `단일 세션 자동 저장 오탐`을 제거하는 안정화 작업이다.

핵심 범위:

1. 동일 노트 자동 저장 요청 직렬화
2. 저장 중 추가 입력 발생 시 최신 draft 재저장 큐잉
3. 단일 세션 빠른 입력에서 `conflict` 오탐 제거
4. 기존 다중 탭/세션 충돌 UX 유지

## 2. 사용자 시나리오

1. 사용자가 같은 노트에서 빠르게 계속 타이핑해도 `덮어쓰기/취소` 팝업이 뜨지 않는다.
2. 자동 저장이 진행 중인 동안 추가 입력이 발생해도, 이전 저장 완료 후 최신 내용이 다시 저장된다.
3. 사용자가 입력 중 노트 이동을 시도하면 기존 `저장 후 이동 / 버리고 이동 / 취소` 흐름은 그대로 유지된다.
4. 실제로 다른 탭이나 세션에서 같은 노트를 수정한 경우에는 기존처럼 `conflict` 상태와 `덮어쓰기/취소` 선택 UI가 떠야 한다.

## 3. 문제 정의

현재 구현은 자동 저장 시마다 `updated_at`을 함께 보내고, 서버는 최신 `updated_at`과 다르면 `409 CONFLICT`를 반환한다. 이 계약 자체는 `FEATURE-004`와 `PRD SR-9` 기준으로 맞다.

문제는 클라이언트가 같은 노트에 대한 저장 요청을 직렬화하지 않는다는 점이다.

현재 구조:

1. 입력 변경 시 800ms 디바운스로 자동 저장 예약
2. 저장 요청이 시작되면 현재 `selectedNoteUpdatedAtRef.current`를 기준으로 `PATCH /api/notes/:id` 호출
3. 저장이 끝나기 전에 다음 자동 저장이 출발할 수 있음
4. 뒤늦게 출발한 요청이 이전 `updated_at` 값을 들고 가면 서버가 정상적으로 `409 CONFLICT` 반환
5. 프런트는 이를 `다른 세션 충돌`로 해석해 팝업 표시

즉, 서버 계약 오류가 아니라 `같은 세션 내부 요청 경합`을 `실제 충돌`로 오인하는 클라이언트 상태 관리 문제다.

## 4. 기능 요구사항

1. 같은 노트에 대해 동시에 두 개 이상의 저장 요청이 진행되면 안 된다.
2. 저장 중 추가 입력이 발생하면 두 번째 요청을 즉시 보내지 말고 `최신 draft 다시 저장 필요` 상태로만 기록해야 한다.
3. 현재 저장이 성공하면, 저장 시작 시점 이후 draft가 바뀌었는지 확인하고 바뀌었다면 최신 `updated_at` 기준으로 즉시 다음 저장을 이어서 실행해야 한다.
4. 단일 세션 빠른 입력만으로 `conflict` 팝업이 발생하면 안 된다.
5. 실제 다중 탭/세션 충돌은 기존처럼 `conflict` 상태와 `덮어쓰기/취소` UI로 처리되어야 한다.
6. 사용자에게 보이는 저장 상태 모델 `dirty/saving/saved/error/conflict`는 유지하되, 요청 직렬화를 위한 내부 상태는 별도 ref/flag로 관리한다.
7. 저장 응답이 늦게 도착해도 현재 선택된 노트가 바뀐 경우 다른 노트 상태를 오염시키면 안 된다.

## 5. API/데이터 계약

1. 1차 수정에서는 서버 API 계약을 바꾸지 않는다.
2. 기존 `PATCH /api/notes/:id` 요청의 `updated_at`, `force` 계약을 그대로 사용한다.
3. 서버의 `409 CONFLICT`는 계속 `실제 최신본과 요청 기준 시점이 다르다`는 의미로 유지한다.
4. 이번 수정은 `409`의 의미를 바꾸는 것이 아니라, 클라이언트가 불필요한 stale 요청을 만들지 않도록 조정하는 데 집중한다.

## 6. 상태/예외 처리

권장 내부 상태:

1. `saveInFlightRef`
- 현재 선택 노트에 대한 저장 요청 진행 여부

2. `queuedSaveRef`
- 저장 중 draft가 다시 바뀌어 후속 저장이 필요한지 여부

3. `lastRequestedDraftRef`
- 현재 요청에 실어 보낸 title/content 스냅샷

동작 원칙:

1. 입력 이벤트는 항상 최신 draft를 state/ref에 반영한다.
2. 저장 중 입력이 발생하면 `queuedSaveRef = true`만 설정하고, 새 요청은 보내지 않는다.
3. 현재 요청 성공 후 draft가 이미 더 바뀌어 있으면 곧바로 다음 저장을 시작한다.
4. `conflict` 상태에서는 자동 저장 재시작을 막고, 명시적 사용자 선택 이후에만 `force` 저장을 허용한다.
5. `error` 상태에서 재시도할 때도 동일 직렬화 규칙을 탄다.

## 7. 권장 구현 순서

### Task 1. 자동 저장 상태 머신 정리

주요 작업:

1. `NotesPage` 내부 자동 저장 관련 ref/flag를 명시적으로 분리
2. `save in flight`와 `queued save`를 구분
3. 기존 `saveStatus`는 사용자 표시 상태로만 유지

완료 기준:

1. 저장 요청 경합 제어 로직이 한 경로로 모인다.

### Task 2. `persistCurrentNote` 직렬화

주요 작업:

1. 저장 중 추가 요청 발생 시 즉시 API를 보내지 않도록 변경
2. 저장 성공 후 최신 draft 차이를 비교해 후속 저장을 이어서 실행
3. 현재 노트가 바뀐 경우 응답 적용 범위를 안전하게 제한

완료 기준:

1. 같은 노트에서 동시 저장 요청이 발생하지 않는다.
2. 빠른 입력 후에도 최신 draft가 누락 없이 저장된다.

### Task 3. 전환/재시도 경로 정합성 점검

주요 작업:

1. `저장 후 이동`, `버리고 이동`, `다시 저장` 흐름이 직렬화 구조와 충돌하지 않는지 검증
2. `conflict` 상태 진입 조건이 실제 stale 요청에 한정되는지 점검

완료 기준:

1. 기존 전환 보호 UX가 깨지지 않는다.

### Task 4. 테스트 보강

주요 작업:

1. 빠른 연속 입력 + 자동 저장 경합 회귀 테스트 추가
2. 저장 중 추가 입력 후 후속 저장이 이어지는지 검증
3. 실제 다중 세션 충돌 테스트가 여전히 유지되는지 확인

완료 기준:

1. 단일 세션 오탐은 사라지고, 실제 충돌 테스트는 계속 통과한다.

## 10. 연계 문서 고려 사항

1. `PRD SR-9` 기준으로 충돌 UI는 `실제 동시 수정 충돌`에만 사용되어야 한다.
2. `DESIGN_GUIDELINES` 기준으로 저장 전/후 상태는 계속 명확해야 하며, 실패/충돌 상태를 오남용하면 안 된다.
3. `DESIGN_SYSTEM` 기준으로 `dirty/saving/saved/error/conflict` 사용자 표시 모델은 유지하되 내부 제어 상태와 분리하는 편이 맞다.
4. `COMPONENT_GUIDE` 기준으로 저장 상태는 사용자에게 노출 가능한 형태로 유지하고, 내부 요청 큐 상태는 UI에 직접 노출하지 않는 것이 적절하다.

## 11. 오픈 이슈

1. 자동 저장 직렬화 로직을 `NotesPage` 내부에 유지할지, 별도 훅/유틸로 분리할지 결정이 필요하다.
2. 회귀 테스트를 컴포넌트 수준으로 넣을지, 훅/유틸 분리 후 단위 테스트 중심으로 갈지 결정이 필요하다.
