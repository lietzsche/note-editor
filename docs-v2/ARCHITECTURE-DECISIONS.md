# 아키텍처 결정 기록 (ADR)

## 개요

이 문서는 노트 편집기 V2 프로젝트의 중요한 아키텍처 결정을 기록합니다. 각 결정은 문제, 고려사항, 결정, 결과를 포함하여 기록됩니다.

## ADR 템플릿

각 ADR은 다음 형식을 따릅니다:

```
# ADR-{번호}: {제목}

## 상태
{제안됨|채택됨|폐기됨|대체됨}

## 날짜
{YYYY-MM-DD}

## 의사 결정자
{이름/역할}

## 문제
{해결해야 할 문제 설명}

## 고려사항
### 옵션 1: {옵션 이름}
- 장점:
- 단점:

### 옵션 2: {옵션 이름}
- 장점:
- 단점:

## 결정
{선택한 옵션 및 이유}

## 결과
{결정의 결과 및 영향}
```

## 결정 목록

### ADR-001: 프론트엔드 프레임워크 선택

**상태**: 채택됨  
**날짜**: 2026-04-21  
**의사 결정자**: 기술 리더십 팀  

**문제**: V2 프로젝트의 프론트엔드 프레임워크를 선택해야 합니다. V1에서는 React를 사용했으나, 다른 옵션도 고려해야 합니다.

**고려사항**:
#### 옵션 1: React 18 유지
- 장점:
  - 팀의 기존 경험 활용 가능
  - 대규모 생태계 및 커뮤니티 지원
  - 서버 컴포넌트 등 최신 기능 활용
- 단점:
  - 번들 크기가 상대적으로 큼
  - 학습 곡선 존재

#### 옵션 2: Vue 3로 전환
- 장점:
  - 점진적 채용 가능
  - Composition API로 더 나은 타입 지원
  - 더 작은 번들 크기
- 단점:
  - 팀 재교육 필요
  - 생태계 규모가 React보다 작음

#### 옵션 3: SolidJS 채용
- 장점:
  - 뛰어난 성능
  - 작은 번들 크기
  - 반응성 시스템이 직관적
- 단점:
  - 상대적으로 새로운 기술
  - 생태계가 성장 중
  - 팀 재교육 필요

**결정**: 옵션 1 (React 18 유지) 채택  
**이유**:
1. 팀의 기존 전문성 활용 가능
2. V1 코드베이스와의 연속성 유지
3. React 생태계의 안정성과 성숙도
4. 서버 컴포넌트 등 차세대 기능 활용 가능

**결과**:
- React 18, TypeScript, Vite로 기술 스택 결정
- 함수형 컴포넌트와 Hooks 사용
- 상태 관리는 Context API + useReducer로 시작

### ADR-002: 백엔드 프레임워크 선택

**상태**: 채택됨  
**날짜**: 2026-04-21  
**의사 결정자**: 백엔드 아키텍트  

**문제**: Cloudflare Workers 환경에서 실행될 백엔드 프레임워크를 선택해야 합니다.

**고려사항**:
#### 옵션 1: Hono 채택
- 장점:
  - 경량화된 디자인 (최소 런타임 오버헤드)
  - Cloudflare Workers에 최적화
  - TypeScript 퍼스트 클래스 지원
  - 미들웨어 및 라우팅 시스템
- 단점:
  - 상대적으로 새로운 프레임워크
  - 생태계가 아직 성장 중

#### 옵션 2: Express.js + Cloudflare Workers Adapter
- 장점:
  - 익숙한 API와 대규모 생태계
  - 풍부한 미들웨어 라이브러리
  - 검증된 솔루션
- 단점:
  - Node.js 런타임에 의존
  - Workers 환경에서의 오버헤드
  - Cold start 시간 증가 가능성

#### 옵션 3: Native Cloudflare Workers API
- 장점:
  - 최소한의 오버헤드
  - Workers 환경에 완전 최적화
  - 공식 문서와의 일관성
- 단점:
  - 저수준 API
  - 반복적인 보일러플레이트 코드
  - 프레임워크 기능 부재

**결정**: 옵션 1 (Hono 채택)  
**이유**:
1. Cloudflare Workers에 특화된 디자인
2. 우수한 TypeScript 지원
3. Express와 유사한 친숙한 API
4. 경량화된 런타임

**결과**:
- Hono를 기본 백엔드 프레임워크로 채택
- 표준화된 에러 처리 및 미들웨어 체계 구축

### ADR-003: 데이터베이스 선택

**상태**: 채택됨  
**날짜**: 2026-04-21  
**의사 결정자**: 데이터 아키텍트  

**문제**: Cloudflare 환경에서 실행될 데이터베이스 솔루션을 선택해야 합니다.

**고려사항**:
#### 옵션 1: Cloudflare D1 (SQLite)
- 장점:
  - Cloudflare 네이티브 통합
  - SQLite 호환성 (익숙한 SQL)
  - 자동 복제 및 백업
  - 지역별 배포 가능
- 단점:
  - 아직 베타 단계
  - 확장성 제한 가능성
  - 고급 기능 제한

#### 옵션 2: PostgreSQL + 연결 풀
- 장점:
  - 강력한 기능 세트
  - 검증된 안정성
  - 풍부한 생태계
- 단점:
  - 외부 종속성 증가
  - 연결 지연 시간
  - 관리 복잡도

#### 옵션 3: Cloudflare KV (Key-Value)
- 장점:
  - 초고속 읽기 성능
  - 글로벌 분산
  - 간단한 API
- 단점:
  - 관계형 데이터에 부적합
  - 쿼리 기능 제한
  - 강력한 일관성 부족

**결정**: 옵션 1 (Cloudflare D1) 채택  
**이유**:
1. Cloudflare 생태계의 완전한 통합
2. SQLite의 익숙함과 유연성
3. 서버리스 아키텍처와의 호환성
4. 관리 부담 감소

**결과**:
- D1을 기본 관계형 데이터베이스로 채택
- 마이그레이션 시스템 구축
- 백업 및 복구 전략 수립

### ADR-004: 인증 방식 선택

**상태**: 채택됨  
**날짜**: 2026-04-21  
**의사 결정자**: 보안 아키텍트  

**문제**: 다중 사용자 시스템의 인증 및 권한 부여 방식을 결정해야 합니다.

**고려사항**:
#### 옵션 1: JWT (JSON Web Tokens)
- 장점:
  - 상태 비저장 (Stateless)
  - 확장성 우수
  - 모바일/웹 호환성
  - 표준 기반
- 단점:
  - 토큰 폐기 어려움
  - 토큰 크기 제한
  - 클라이언트 측 저장 필요

#### 옵션 2: 세션 기반 인증
- 장점:
  - 토큰 폐기 용이
  - 세부적인 세션 관리
  - 서버 측 제어
- 단점:
  - 상태 저장 필요
  - 확장성 제한
  - 쿠키 의존성

#### 옵션 3: OAuth 2.0 + OpenID Connect
- 장점:
  - 표준화된 프로토콜
  - 제3자 인증 통합
  - 안전한 위임
- 단점:
  - 구현 복잡성
  - 추가 인프라 필요
  - 학습 곡선

**결정**: 하이브리드 접근 방식 (JWT + 짧은 만료 + 리프레시 토큰)  
**이유**:
1. JWT의 상태 비저장 이점 활용
2. 짧은 액세스 토큰 만료로 보안 강화
3. 리프레시 토큰으로 사용자 경험 유지
4. 필요시 세션 무효화 메커니즘 구현

**결과**:
- JWT 기반 인증 시스템 구현
- 액세스 토큰 (15분), 리프레시 토큰 (7일) 정책
- 토큰 블랙리스트 메커니즘 추가

### ADR-005: 상태 관리 전략

**상태**: 채택됨  
**날짜**: 2026-04-21  
**의사 결정자**: 프론트엔드 아키텍트  

**문제**: React 애플리케이션의 상태 관리 전략을 결정해야 합니다.

**고려사항**:
#### 옵션 1: Context API + useReducer
- 장점:
  - React 내장 솔루션
  - 추가 의존성 없음
  - 학습 곡선 낮음
  - 작은 앱에 적합
- 단점:
  - 성능 최적화 필요
  - 복잡한 상태에 부적합
  - 보일러플레이트 코드

#### 옵션 2: Zustand
- 장점:
  - 경량화된 설계
  - 간단한 API
  - 미들웨어 지원
  - TypeScript 퍼스트 클래스
- 단점:
  - 추가 의존성
  - 커뮤니티 규모 작음

#### 옵션 3: Redux Toolkit
- 장점:
  - 예측 가능한 상태 관리
  - 강력한 개발자 도구
  - 대규모 생태계
  - 검증된 솔루션
- 단점:
  - 상당한 보일러플레이트
  - 학습 곡선
  - 과도한 엔지니어링 가능성

**결정**: 점진적 접근 방식  
**이유**:
1. 작은 규모로 시작하여 복잡도 증가시 확장
2. 초기에는 Context API + useReducer로 시작
3. 상태 관리 복잡도 증가시 Zustand로 전환 계획
4. 불필요한 초기 복잡도 회피

**결과**:
- 애플리케이션 상태를 도메인별로 분리
- 사용자, 노트, 그룹별 Context 생성
- 복잡도 임계점 도달시 Zustand 도입 준비

### ADR-006: 스타일링 방법 선택

**상태**: 채택됨  
**날짜**: 2026-04-21  
**의사 결정자**: UI/UX 아키텍트  

**문제**: 컴포넌트 스타일링 및 디자인 시스템 구현 방법을 결정해야 합니다.

**고려사항**:
#### 옵션 1: CSS Modules
- 장점:
  - 지역 범위 CSS
  - 간단한 학습 곡선
  - 빌드 도구 통합
  - 성능 우수
- 단점:
  - 동적 스타일링 제한
  - 테마 지원 복잡
  - 글로벌 스타일 관리 어려움

#### 옵션 2: Styled Components
- 장점:
  - CSS-in-JS 유연성
  - 동적 스타일링 용이
  - 테마 지원 우수
  - 컴포넌트 기반 설계
- 단점:
  - 런타임 오버헤드
  - 번들 크기 증가
  - 서버 사이드 렌더링 복잡

#### 옵션 3: Tailwind CSS
- 장점:
  - 유틸리티 퍼스트 접근
  - 일관된 디자인 시스템
  - 작은 번들 크기
  - 빠른 개발 속도
- 단점:
  - 학습 곡선
  - 클래스 이름 장황함
  - 디자인 의사결정 제한

**결정**: CSS Modules + 디자인 토큰 시스템  
**이유**:
1. 성능과 유지보수성의 균형
2. 디자인 토큰으로 일관된 테마 관리
3. CSS 변수를 활용한 다크 모드 지원
4. 점진적 개선 가능

**결과**:
- CSS Modules를 기본 스타일링 방법으로 채택
- 디자인 토큰 JSON 파일로 색상, 타이포그래피 정의
- CSS 변수를 활용한 테마 전환 시스템 구축

### ADR-007: 테스트 전략

**상태**: 채택됨  
**날짜**: 2026-04-21  
**의사 결정자**: QA 아키텍트  

**문제**: 프로젝트의 테스트 전략과 도구를 결정해야 합니다.

**고려사항**:
#### 옵션 1: Vitest + React Testing Library
- 장점:
  - Vite 생태계 통합
  - 빠른 실행 속도
  - 모의(mocking) 지원 우수
  - TypeScript 퍼스트 클래스
- 단점:
  - 상대적으로 새로운 도구
  - 커뮤니티 성장 중

#### 옵션 2: Jest + React Testing Library
- 장점:
  - 업계 표준
  - 대규모 생태계
  - 풍부한 문서
  - 검증된 안정성
- 단점:
  - 설정 복잡성
  - 실행 속도 상대적으로 느림
  - Vite와의 통합 어려움

#### 옵션 3: Playwright (E2E) + Vitest (단위)
- 장점:
  - 통합 테스트 커버리지
  - 크로스 브라우저 지원
  - 현실적인 사용자 시나리오
- 단점:
  - 유지보수 비용 높음
  - 실행 속도 느림
  - 설정 복잡성

**결정**: 계층적 테스트 접근 방식  
**이유**:
1. Vitest + React Testing Library로 단위 및 통합 테스트
2. Cloudflare Workers 테스트 환경으로 백엔드 테스트
3. 필요시 Playwright로 E2E 테스트 추가
4. 테스트 피라미드 원칙 준수

**결과**:
- Vitest를 기본 테스트 러너로 채택
- React Testing Library로 컴포넌트 테스트
- Cloudflare Workers 테스트 환경 구성
- 테스트 커버리지 목표 설정 (핵심 기능 80%)

## 검토 및 업데이트 프로세스

### 정기 검토
- 모든 ADR은 6개월마다 검토
- 기술 변화나 요구사항 변경시 재평가
- 폐기된 ADR은 보관용으로 유지

### 새로운 ADR 생성
1. 아키텍처 결정 필요시 ADR 템플릿 사용
2. 관련 팀원과 검토
3. 기술 리더십 승인
4. 문서화 및 커밋

### ADR 수정
1. 기존 ADR의 상태를 "대체됨"으로 변경
2. 새로운 ADR 생성 (참조 포함)
3. 변경 이유 및 영향 문서화
