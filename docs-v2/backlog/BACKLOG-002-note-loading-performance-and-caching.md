## 0. 이번 차수 결론

이번 차수에서는 저위험 성능 개선 범위를 우선 마감했다.

1. 개발 환경에서 그룹 전환/노트 열기 시간을 직접 확인할 수 있는 최소 계측 지점을 추가했다.
2. 그룹별 노트 목록 메모리 캐시와 `stale-while-revalidate` 흐름을 도입했다.
3. 생성/수정/삭제/이동/정렬 이후 현재 화면 우선 갱신과 캐시 무효화/부분 갱신 규칙을 반영했다.
4. `NotesPage` 대규모 분리와 목록/상세 API 분리는 이번 1차 마감 범위에서 제외하고, 실제 체감 수치가 추가로 필요할 때 후속 backlog로 다시 판단한다.
5. 수동 cold/warm 측정값 기록은 `TEST-BACKLOG-002` 기준으로 이어서 남길 수 있게 유지한다.

## 1. 문제 정의

현재 프로젝트는 노트나 그룹을 선택할 때 체감 지연이 발생한다는 사용성 문제가 있다.

특히 다음 흐름이 느리게 느껴질 수 있다.

1. 그룹 클릭 후 해당 그룹 노트가 보이기까지의 시간
2. 노트 클릭 후 편집기 내용이 안정적으로 보이기까지의 시간
3. 생성/삭제/이동 이후 목록이 다시 반영되기까지의 시간

## 2. 현재 구현에서 확인된 원인 후보

현재 코드 기준으로 다음 특성이 확인된다.

1. 그룹 선택 시마다 `loadNotes(selectedGroupId)`가 다시 호출되어 네트워크 재조회가 발생한다.
2. 노트 목록 조회 API가 목록용 요약이 아니라 `content`를 포함한 전체 노트 데이터를 내려준다.
3. 생성/삭제/그룹 이동 후에도 전체 목록을 다시 조회하는 흐름이 많다.
4. 그룹/노트/에디터 상태가 `NotesPage`에 집중되어 있어 큰 상태 변경 시 재렌더 부담이 커질 수 있다.
5. 그룹/노트 목록에 대한 메모리 캐시, stale-while-revalidate, 요청 중복 제거가 없다.

추가 해석:

1. 현재 노트 클릭은 별도 상세 API를 다시 호출하지 않고, 목록에 이미 담긴 `content`로 바로 `openNote(note)`를 수행한다.
2. 따라서 `노트 클릭이 느리다`는 체감은 네트워크보다 큰 본문 state 전환, textarea 렌더링, 목록/편집 패널 동시 재렌더에서 오는 가능성이 높다.
3. 반대로 `그룹 클릭이 느리다`는 체감은 목록 재조회와 큰 payload 전송의 영향이 실제로 클 수 있다.

## 2.1 문서 교차 검토 결과

### 기능 문서 기준

1. `FEATURE-002`는 현재 목록 조회와 정렬 유지에 초점을 두고 있고, 성능 기준은 대량 정렬 이슈만 열려 있다.
2. `FEATURE-003`는 그룹 필터 정확성에 초점을 두고 있으며, 그룹 선택 시 빠른 재탐색 경험 자체는 아직 별도 규격화되지 않았다.

### 디자인 문서 기준

1. `DESIGN_GUIDELINES`의 `연속성` 원칙상 목록 -> 편집 -> 목록 복귀의 인지 부하는 낮아야 한다.
2. 같은 문서의 `안정성` 원칙상 전환 중 빈 화면이나 늦은 치환보다 예측 가능한 상태 표시가 우선이다.
3. `DESIGN_SYSTEM` 기준으로는 `loading/ready/error` 상태를 일관되게 표현해야 하고, `dirty/conflict`를 캐시 갱신이 덮어써서는 안 된다.
4. `COMPONENT_GUIDE` 기준으로는 fetch/state orchestration을 컨테이너에 두되, 화면 컨테이너와 렌더링 컴포넌트를 분리하는 것이 권장된다.

정리:

1. 성능 개선은 단순 API 캐시만이 아니라 `상태 전환 UX`, `렌더 분리`, `무결성 보호`를 같이 봐야 한다.
2. 캐시 hit 시에도 현재 편집 중인 노트의 `dirty/saving/conflict` 상태는 백그라운드 갱신보다 우선되어야 한다.

## 3. 검토 목표

1. 반복 방문하는 그룹과 노트는 더 빠르게 보여야 한다.
2. 캐시를 도입하더라도 편집 무결성과 최신성 정책이 깨지면 안 된다.
3. 사용자에게는 빈 화면보다 `즉시 표시 + 백그라운드 갱신`이 우선되어야 한다.
4. 저위험 개선부터 단계적으로 적용할 수 있어야 한다.

## 3.1 1차 권장안

현재 코드와 문서 기준으로는 다음 순서가 가장 안전하다.

1. 그룹 목록/그룹별 노트 목록에 메모리 캐시를 도입한다.
2. 그룹 전환은 `캐시 즉시 표시 + 백그라운드 재검증`으로 바꾼다.
3. 생성/삭제/이동 후 전체 재조회 대신 낙관적 목록 갱신 + 선택적 무효화를 우선 적용한다.
4. 노트 클릭 지연은 별도 detail fetch보다 `렌더 분리`와 `불필요한 상태 전파 감소`로 먼저 본다.
5. 목록/상세 API 분리는 계측 결과상 payload 병목이 확실할 때 2차로 검토한다.

권장 이유:

1. 현재 노트 클릭은 네트워크 병목이 아닐 가능성이 높다.
2. 따라서 바로 API를 쪼개기보다, `그룹 전환 캐시`와 `렌더 비용 절감`이 선행되는 편이 리스크가 낮다.

## 4. 검토 가능한 대응안

### 4.1 프론트엔드 메모리 캐시

1. 그룹 목록 캐시
2. 그룹별 노트 목록 캐시
3. 노트 상세 캐시

장점:

1. 구현 리스크가 비교적 낮다.
2. 같은 세션 내 재방문 속도를 빠르게 만들 수 있다.

주의점:

1. 생성/수정/삭제/이동 후 캐시 무효화 정책이 필요하다.

### 4.2 Stale-While-Revalidate 패턴

1. 캐시된 데이터가 있으면 먼저 즉시 렌더링
2. 백그라운드에서 최신 데이터를 다시 조회
3. 차이가 있으면 조용히 상태 갱신

장점:

1. 그룹 재선택 시 체감 속도 개선 효과가 크다.
2. 완전한 오프라인 캐시보다 구현 난도가 낮다.

### 4.3 목록/상세 API 분리

1. 목록 API는 `id`, `title`, `group_id`, `sort_order`, `updated_at`, 선택적 snippet만 반환
2. 본문 `content`는 상세 조회 시 불러오거나 미리 예열(prefetch)한다.

장점:

1. 그룹 전환 시 payload를 크게 줄일 수 있다.
2. 많은 노트/긴 본문 환경에서 효과가 크다.

주의점:

1. 노트 클릭 시 detail fetch가 추가되면 오히려 느려질 수 있다.
2. 따라서 상세 캐시 또는 prefetch와 함께 검토해야 한다.
3. 현재는 노트 클릭 시 추가 fetch가 없으므로, 이 대응안은 1차 개선보다는 2차 검토안에 가깝다.

### 4.4 낙관적 목록 업데이트

1. 생성/삭제/이동/정렬 후 전체 재조회 전에 로컬 상태를 먼저 갱신
2. 서버 응답 실패 시 롤백

장점:

1. 조작 직후 반응성이 좋아진다.

### 4.5 계측 및 성능 기준 수립

1. 그룹 전환 시간
2. 노트 열기 시간
3. 캐시 hit/miss 비율
4. 노트 수/본문 크기에 따른 payload 크기

장점:

1. 체감 이슈를 정량화하고 우선순위를 잡을 수 있다.

## 6. 수용 기준

1. 같은 세션에서 이미 본 그룹은 재선택 시 더 빠르게 보여야 한다.
2. 노트 선택 시 빈 본문이나 늦은 본문 치환 같은 어색한 지연이 줄어들어야 한다.

4. 생성/수정/삭제/이동 후 오래된 캐시가 잘못 남지 않아야 한다.
5. 개선 전/후 비교가 가능한 성능 기록이 남아야 한다.
