# Note Editor Code Review

## 목적

코드를 요약하는 것이 아니라 실제 문제가 될 수 있는 버그와 회귀 위험을 찾는다. 사용자가 “리뷰해줘”, “PR 봐줘”, “이 변경 괜찮아?”라고 하면 이 skill을 사용한다.

## 먼저 확인할 것

1. `git diff`, 변경 파일, 관련 테스트를 확인한다.
2. 기능 의도가 불분명하면 관련 `docs/product/PRD.md`, `docs/features/`, `docs/testing/TEST_PLAN.md`를 확인한다.
3. 데이터 흐름이 있는 변경은 클라이언트 상태, API 계약, D1 스키마를 함께 추적한다.

## 우선순위

1. 데이터 손실, 노트 내용 섞임, 자동 저장 오동작, 충돌 처리 오류.
2. 계정 격리, 인증/세션, 공유 링크 접근 제어 문제.
3. all-notes/group-scoped cache 동기화, reorder persistence 문제.
4. API envelope, status code, client wrapper 불일치.
5. 테스트 누락과 수용 기준 미검증.
6. 유지보수성, 중복, 타입 안정성, 네이밍 문제.

## 리뷰 방식

1. findings를 먼저 쓴다. 요약은 뒤로 보낸다.
2. 각 finding에는 파일과 라인을 포함한다.
3. 사용자 영향과 재현 가능성을 설명한다.
4. 확실하지 않으면 “추정”이라고 표시하고 확인 방법을 적는다.
5. 단순 취향이나 스타일 차이는 회귀 위험이 있을 때만 지적한다.
6. 리뷰 요청이면 코드를 수정하지 않는다. 사용자가 “수정해줘”라고 하면 apply-feedback skill로 전환한다.

## 출력 포맷

- `High`: 데이터 손실, 보안, 주요 기능 실패 가능성.
- `Medium`: 특정 플로우 회귀, 계약 불일치, 테스트로 잡아야 할 문제.
- `Low`: 유지보수성, 일관성, 작은 UX/구조 문제.
- `Open questions`: 의도 확인이 필요한 항목.
- `Residual risk`: 리뷰 후에도 남는 검증 공백.

## 예시 요청

- “최근 변경사항 코드 리뷰해줘.”
- “공유 링크 구현 PR에서 보안 문제 없는지 봐줘.”
- “자동 저장 수정한 거 회귀 위험 위주로 리뷰해줘.”
