# Note Editor Design Brainstorm

## 목적

정답 하나를 바로 구현하지 않고 여러 디자인 방향을 탐색한다. 사용자가 “어떻게 보이면 좋을까”, “다른 UX 없을까”, “디자인 아이디어 줘”처럼 요청하면 선택 가능한 대안을 만든다.

## 먼저 확인할 것

1. `docs/design/DESIGN_SYSTEM.md`에서 상태/피드백/접근성 원칙을 확인한다.
2. `docs/design/DESIGN_GUIDELINES.md`, `docs/design/COMPONENT_GUIDE.md`, `docs/design/THEME_TOKENS.md`를 필요한 만큼 확인한다.
3. 현재 화면 구조가 중요하면 `src/components/NotesPageLayout.tsx`, `src/pages/NotesPage.tsx`, `src/pages/notesPageStyles.ts`를 확인한다.

## 작업 방식

1. 현재 사용자의 문제나 감정, 작업 맥락을 한 문장으로 정의한다.
2. 최소 2개, 많아도 4개 정도의 디자인 방향을 제안한다.
3. 각 방향마다 장점, 단점, 구현 난이도, 회귀 위험을 짧게 비교한다.
4. 모바일과 데스크톱에서 흐름이 달라지는지 확인한다.
5. 접근성, 키보드 조작, 상태 피드백을 아이디어 단계에서도 포함한다.
6. 사용자가 구현을 요청하기 전에는 코드를 수정하지 않는다.

## 대안 제시 포맷

- 방향 이름.
- 핵심 아이디어.
- 잘 맞는 상황.
- 주의할 점.
- 구현 난이도.
- 추천 여부.

## 디자인 원칙

- 기존 디자인 시스템과 토큰을 우선한다.
- 노트 작성 흐름을 방해하는 장식은 피한다.
- 저장 상태, 충돌 상태, 복사 결과, reorder 상태는 색상만이 아니라 텍스트로도 전달한다.
- 모바일에서는 그룹, 노트 목록, 에디터 패널 전환 비용을 낮추는 방향을 우선한다.

## 예시 요청

- “노트 목록 빈 상태 UI를 좀 더 좋게 만들 아이디어를 줘.”
- “공유 링크 UI를 어떤 식으로 배치하면 좋을지 3가지 방향으로 브레인스토밍해줘.”
- “모바일에서 그룹과 노트 전환 UX를 개선할 방법을 제안해줘.”
