# Note Editor Design Review

## 목적

디자인을 칭찬하거나 설명하는 것이 아니라 사용성 문제, 접근성 문제, 상태 누락, 디자인 시스템 이탈, 구현 시 회귀 위험을 찾는다.

## 먼저 확인할 것

1. 변경된 화면이나 제안서를 확인한다.
2. `docs/design/` 문서에서 토큰, 컴포넌트, 접근성 기준을 확인한다.
3. 관련 기능 문서에서 사용 시나리오와 수용 기준을 확인한다.
4. 코드 구현이 있다면 UI 구조와 상태 전달 방식을 확인한다.

## 리뷰 관점

- 사용자가 다음 행동을 이해할 수 있는가.
- loading, empty, error, saving, saved, conflict 상태가 명확한가.
- 모바일과 데스크톱에서 주요 플로우가 끊기지 않는가.
- 키보드 조작과 focus 표시가 유지되는가.
- 색상만으로 상태를 전달하지 않는가.
- 터치 타깃과 클릭 영역이 충분한가.
- 기존 디자인 토큰과 컴포넌트 패턴을 벗어나지 않는가.
- 노트 작성이라는 핵심 작업보다 부가 UI가 더 눈에 띄지 않는가.

## 응답 방식

1. 문제를 심각도 순으로 먼저 쓴다.
2. 각 문제에는 사용자 영향과 수정 방향을 함께 쓴다.
3. 추측이면 추측이라고 표시한다.
4. 구현 요청이 아니라 리뷰 요청이면 코드를 수정하지 않는다.

## 리뷰 출력 포맷

- `High`: 사용자가 작업을 실패하거나 데이터 손실/혼란 가능성이 큰 문제.
- `Medium`: 주요 플로우를 방해하거나 접근성/반응형 품질을 낮추는 문제.
- `Low`: 일관성, 미세 UX, 카피, 정렬, 여백 문제.
- `Open questions`: 판단에 필요한 추가 정보.

## 예시 요청

- “이 공유 UI 디자인 검토해줘.”
- “모바일 노트 편집 화면 UX 문제 있는지 리뷰해줘.”
- “디자인 문서 기준으로 현재 레이아웃이 맞는지 봐줘.”
