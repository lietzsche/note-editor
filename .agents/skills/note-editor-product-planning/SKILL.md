# Note Editor Product Planning

## 목적

구현부터 시작하지 않고 “무엇을 왜 만들지”를 먼저 정리한다. 사용자가 아이디어, 불편함, 기능 제안, 백로그 항목을 말하면 제품 요구사항과 검증 가능한 수용 기준으로 바꾼다.

## 먼저 확인할 것

1. `docs/README.md`에서 현재 canonical 문서를 찾는다.
2. `docs/product/PRD.md`에서 관련 SR과 제품 목표를 확인한다.
3. 관련 기능이 있으면 `docs/features/FEATURE-*.md`와 `docs/backlog/`를 확인한다.
4. 검증 기준이 필요하면 `docs/testing/TEST_PLAN.md`를 확인한다.

## 작업 방식

1. 사용자의 요청을 문제, 사용자, 목표, 성공 기준으로 분해한다.
2. “이번 범위에 포함할 것”과 “이번 범위에서 제외할 것”을 나눈다.
3. 사용자 시나리오를 2~5개 정도로 구체화한다.
4. 수용 기준은 테스트 가능한 문장으로 쓴다.
5. API, DB, UI 같은 구현 세부는 필요한 만큼만 적고, 아직 확정되지 않은 것은 오픈 이슈로 둔다.
6. 구현 요청이 아니라 기획 요청이면 코드는 수정하지 않는다.

## 산출물 형태

- 문제 정의.
- 사용자 시나리오.
- 기능 요구사항.
- 비범위.
- 수용 기준.
- 테스트 관점.
- 오픈 이슈.
- 다음 액션.

## 이 프로젝트에서 특히 볼 것

- 계정 격리, 노트 전환 무결성, 자동 저장 충돌, 그룹/노트 reorder, 공유 링크는 회귀 위험이 높다.
- UI 용어는 사용자에게는 폴더/카테고리식 표현이 가능하지만 문서/API/DB에서는 `group` 추적성을 유지한다.
- 새 기능은 `PRD SR -> FEATURE 문서 -> TEST_PLAN 시나리오 -> 자동/수동 테스트` 흐름으로 연결한다.

## 예시 요청

- “공유 링크에 만료일을 넣고 싶은데 기획부터 정리해줘.”
- “검색 기능을 개선하려는데 요구사항과 수용 기준을 먼저 써줘.”
- “이 백로그를 구현 가능한 feature 문서로 승격해줘.”
