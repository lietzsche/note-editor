# Note Editor Apply Feedback

## 목적

검토나 브레인스토밍에서 나온 제안을 실제 변경으로 반영한다. “이 리뷰 반영해줘”, “2번 방향으로 적용해줘”, “지적된 테스트 누락 수정해줘” 같은 요청에 사용한다.

## 작업 순서

1. 반영할 피드백 목록을 항목별로 분리한다.
2. 각 항목을 코드, 테스트, 문서, 디자인 중 어디에 반영해야 하는지 분류한다.
3. 충돌하거나 모호한 피드백은 구현 전에 짧게 확인한다. 단, 합리적으로 추정 가능하면 진행한다.
4. 데이터 손실, 계정 격리, 자동 저장/충돌, 캐시 동기화에 영향을 주는 항목은 먼저 처리한다.
5. 구현 후 관련 테스트를 추가하거나 갱신한다.
6. 사용자-facing 동작이나 수용 기준이 바뀌면 문서도 갱신한다.
7. 마지막에 반영한 항목과 남은 항목을 구분해 보고한다.

## 반영 기준

- 리뷰 지적이 재현 가능한 버그면 우선 수정한다.
- 디자인 피드백은 디자인 시스템과 접근성 기준을 유지하며 반영한다.
- 브레인스토밍 결과는 선택된 방향만 구현하고, 선택되지 않은 대안은 문서화가 필요할 때만 남긴다.
- 기획 변경은 PRD/FEATURE/TEST 흐름이 어긋나지 않게 반영한다.
- 단순 리팩토링은 동작 변경과 분리할 수 있으면 분리한다.

## 검증 기준

- 클라이언트 로직이면 `npm run typecheck`와 `npm run test:unit`을 우선한다.
- API나 DB가 포함되면 `npm run test:integration`을 실행한다.
- 범위가 넓거나 release-facing이면 `npm run verify`를 실행한다.
- 실행하지 못한 검증은 이유를 명확히 남긴다.

## 예시 요청

- “방금 코드 리뷰에서 나온 High/Medium만 반영해줘.”
- “디자인 리뷰 지적사항 중 접근성 문제를 먼저 수정해줘.”
- “브레인스토밍 2번 방향으로 UI를 실제 구현해줘.”
- “테스트 실패 원인 수정하고 필요한 회귀 테스트도 추가해줘.”
